# 地礦AI雛型 v10（雛型第三版）Implementation Plan

## 0. 給執行者（Claude Code / Fable 5）的說明

- **Baseline 檔案**：`archive/original/地礦AI_雛型_v9.html`（單一 HTML 檔，Tailwind CDN + 原生 JS，無建置流程）
- **輸出檔案**：`archive/original/地礦AI_雛型_v10.html`（複製 v9 後在其上修改，**不要覆蓋 v9**）
- **修改方式**：延續既有慣例，用 Python 字串替換（`str.replace` 或 `re.sub`）針對唯一字串定位修改，每次修改後：
  1. 用 `python3 -c "import re; ..."` 或簡單 parser 檢查 HTML 標籤是否配對（開合 div 數量一致）
  2. 用瀏覽器自動化（如有 Playwright）或至少手動檢查對應頁面能否正常 `showPage()` 切換不報錯
  3. 開 console，確認沒有 JS runtime error
- **完成後**：
  1. 更新 `CHANGELOG.md`、`version-plan.json`（新增 v10 條目，含 `diff_from_previous` 與 `commit_message`）
  2. 更新 `README.md`，讓 `index.html` 對外版本說明對應到 v10（若要正式對外發布，另外把 v10 內容複製為根目錄 `index.html`，這一步請等使用者確認後再做，不要自動覆蓋 `index.html`）
  3. git commit，訊息格式比照既有慣例：`Update geology-ai prototype to v10`
- **語言規範**：所有畫面文字、UI label、程式註解一律使用**繁體中文**（沿用專案既有用語，不要簡體字或英文混雜的新詞彙）
- **不要做的事**：不要修改 v1~v8、v8_會議待辦版；不要更動 `#007d6d` 品牌色、Tailwind CDN、Chart.js/Leaflet.js 既有引用方式

## 3. 詳細任務規格

### T1｜地方政府回報查勘結果要有編輯功能

**現況**（`openSurveyDrawer()`，v9 約 3925 行）：
```js
var isEditable = isGov && (d.status === '待查勘' || d.status === '逾期未查勘');
```
只有狀態為「待查勘」或「逾期未查勘」時，D 角色才能看到可填寫表單；一旦送出（`submitSurveyReport()` 把狀態改成「查勘完成」），之後再打開只會進入唯讀模式（`else` 分支，約 4052 行），沒有任何「編輯」按鈕。

**目標**：D 角色在「查勘完成」狀態下，仍要能重新打開表單編輯已回報的內容。

**修改內容**：
1. 唯讀模式（約 4054-4091 行）中，`currentRole === 'D'` 且 `d.status === '查勘完成'` 時，在既有內容下方加一顆「編輯回報內容」按鈕。
2. 點擊後呼叫既有的可編輯表單渲染邏輯（把 `isEditable` 的判斷條件改成函式參數，例如 `renderSurveyEditForm(d, i)`，唯讀/編輯兩個分支都呼叫同一份表單渲染邏輯，避免程式碼重複）。
3. 編輯表單需要「帶入既有已填寫的值」，目前 `surveyData` 陣列中查勘表單欄位（查勘日期、地段地號、違規情形、回報描述、回報人資訊等）沒有全部欄位對應的資料結構，需要先擴充 `surveyData` 的欄位（在檔案開頭資料定義區塊，搜尋 `surveyData` 的宣告處）以儲存這些欄位值，`submitSurveyReport()` 也要把表單值寫回這些欄位，而不是只更新 `status`/`statusClass`/`reportDate`。
4. **是否無限制可編輯到「已結案」都還能改？** 目前假設：**已結案（`closeSurveyCase()` 之後）不可再編輯**，因為結案通常代表流程正式關閉。這是 [推論]，請確認是否正確；如果需要更嚴謹的規則（例如已進入違規後續處理表就鎖定），要與 T8 一併設計。

**驗收標準**：
- D 角色進入「查勘完成」的案件，能看到「編輯回報內容」按鈕
- 點擊後跳出的表單欄位帶有先前填寫的值（不是空白表單）
- 儲存後列表與詳情頁面即時反映更新後的內容
- B/A、C 角色看到的仍是唯讀畫面，沒有編輯按鈕

### T7｜「移請相關單位續處」區塊移到表單最下面

**現況**：在 `openSurveyDrawer()` 的可編輯表單中（約 3972-3994 行），「移請相關單位續處」checkbox + 「移交資訊」條件顯示區塊，目前位於「是否有違規行為」之後、「回報描述」之前（表單中段）。

**目標**：移到整份表單最下面，方便地方政府人員之後回來單獨編輯這個區塊，不用往上滑找其他欄位。

**修改內容**：
- 把「移請相關單位續處」checkbox 區塊（3973-3976 行）與「移交資訊（條件顯示）」區塊（3979-3994 行）整段搬移到「現場照片」區塊（4020-4025 行）之後、「逾期未查勘區塊」（4028-4044 行）之前，或整份表單最底部、「底部按鈕」（4047-4050 行）之前——**建議放在逾期未查勘區塊之後、底部按鈕之前**，因為逾期未查勘是另一種例外流程，續處資訊應該視為「查勘完成後的補充決定」，放在最後一個實質欄位區塊。
- 純粹搬移 HTML 字串位置，不改欄位內容與 `toggleTransferSection()` 邏輯。

**與 T8 的關聯**：如果 T8 決定把「移交資訊」欄位整個移到「違規行為後續處理」頁面去填寫（而不是在查勘回報表單內），這裡的搬移就只是過渡態，請先看完 T8 再決定要不要做這步搬移，或是直接跳過搬移、改成 T8 的方案。**這兩個任務有先後依賴，建議先確認 T8 的資料流方向再動工。**

**驗收標準**：
- 表單視覺順序：查勘日期/次數 → 地段地號 → 是否有違規行為 → 回報描述 → 回報人資訊 → 備註說明 → 現場照片 → 逾期未查勘 → 移請相關單位續處（含條件顯示的移交資訊）→ 儲存/取消按鈕
- Checkbox 勾選/取消勾選行為不變

### T4｜無須查勘的刪除與「待觀察」標註

**現況**：
- `page-survey-report`（變異點查勘回報）沒有「確認無須查勘」的動作，目前是靠人工請系統廠商工程師從後台刪除資料列
- `page-management-list`（建議監測點位清單）狀態機為：`--`（未建議）→ `建議拍攝`（C 標記）→ B/A 確認「是」→`已確認`；B/A 確認「否」→`僅列管`（`mgmtAction()` 約 3529-3531 行）
- 目前**沒有「待觀察」狀態**，會議紀錄提到的狀態組合是「建議拍攝、待觀察」

**目標**（依會議紀錄direct 解讀）：
1. 當某個點位被「確認無須查勘」時：
   - 從「變異點查勘回報」列表中移除（不再出現在 `survey-report` 的清單）
   - 在「建議監測點位清單」中該點位狀態改標註為「待觀察」，而不是直接刪除

**[推論/待確認] 開放問題**（這裡我需要你確認，因為會直接影響狀態機設計，寫錯會讓之後要改的範圍變大）：

1. **「待觀察」是不是就是現有的「僅列管」改名？** 現有 reject 流程（B/A 按「否」）會把狀態設成「僅列管」，語意上跟「待觀察」很接近（都是「不繼續往下處理，但留著看看」的概念）。我的預設方案是：**把「僅列管」重新命名為「待觀察」**，統一狀態值，不新增第三種狀態，UI 文字全部換成「待觀察」。如果你們業務上「僅列管」跟「待觀察」是兩個不同語意（例如僅列管＝地礦中心主動判斷不用理、待觀察＝因為查勘回報說無須查勘才變成這個狀態），那就要新增獨立的第四種狀態，狀態機會更複雜（要處理誰能把「待觀察」再轉回「建議拍攝」等）。
2. **「確認無須查勘」這個動作，觸發點在哪裡？** 會議紀錄沒有明講這是誰、在哪個畫面按下去的。合理猜測有兩種：
   - (a) 在 `page-survey-report` 的列表操作欄新增一顆「確認無須查勘」按鈕（B/A 權限），點擊後觸發跨模組更新（該筆從 survey-report 移除、對應 management-list 那筆狀態變待觀察）
   - (b) 這其實是「地礦中心審查 AI 判釋結果」階段就要決定的事，跟查勘回報無關，應該放在 `page-ai-list` 或 `page-management-list`
   
   目前我先用方案 (a) 寫規格（因為會議紀錄的語境是接在「地方政府回報查勘結果」之後），但**這點務必跟阿寶再確認一次**，因為兩種方案對應的程式改動位置完全不同。
3. **「從變異點查勘回報列表刪除」是真的從資料陣列 `splice` 掉，還是只是換一個「已排除」狀態、預設篩選隱藏？** 雛型階段建議用**狀態隱藏**而非真的刪除資料（例如加一個 `excluded: true` 的欄位，`renderSurveyTable()` 預設過濾掉），這樣之後如果要串資料庫，邏輯比較不會跟「軟刪除 vs 硬刪除」的實務需求衝突，也方便你在demo時視需要切換顯示。

### T8｜違規行為後續處理：動態化、D 角色可見、移交資訊條件顯示

**現況**：`page-urge-violation-followup`（約 1396-1471 行）目前是**完全靜態**的頁面：
- 篩選欄位存在但沒有接任何 JS（搜尋、查詢按鈕沒有 `onclick`）
